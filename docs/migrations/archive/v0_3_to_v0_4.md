# Migration Guide: v0.3.x ŌåÆ v0.4.0

**Release date**: 2026-04-13
**Type**: Breaking change (`fixture.schema.json` ŌĆö `covers` now required)
**Affected consumers**: claude-nexus, opencode-nexus, nexus-code

---

## Summary

v0.4.0 makes the conformance layer self-auditing. `fixture.schema.json` now requires a `covers` field on every fixture object. Consumers that maintain custom fixture sets must add `covers` to every fixture before upgrading. The `validate:conformance` script enforces this at runtime; CI pipelines must add it as a release gate.

Additionally, a new `conformance/lifecycle/` directory introduces 5 event-based fixtures covering harness-managed session and agent lifecycle events. Adoption of lifecycle fixtures is optional but recommended.

---

## Breaking Change: `covers` required in `fixture.schema.json`

### What changed

The top-level `covers` field has been promoted from optional to required in `conformance/schema/fixture.schema.json`. The `uncovered_params` field is new and optional.

Additionally, the schema now uses a `oneOf` branch at the top level to distinguish tool fixtures (which have an `action` key) from event fixtures (which have an `event` key). A fixture object may not have both.

### Why

Without `covers`, there is no machine-verifiable link between a fixture and the schema fields it exercises. Gaps in field coverage were invisible. The `validate:conformance` script can now cross-reference every `state-schema` field against the union of all `covers` declarations and report uncovered fields explicitly.

### Consumer action required

Every fixture file you maintain ŌĆö whether in `conformance/tools/`, `conformance/scenarios/`, or a consumer-local directory ŌĆö must include a `covers` field. If you have no custom fixtures, no action is required beyond adding the CI gate (see below).

## Required Consumer Actions

### 1. Add `covers` to all existing fixtures

`covers` declares which state-schema fields and which return-value fields the fixture exercises. At least one non-empty key (`state_schemas` or `return_value`) must be present.

**Before (v0.3.x fixture ŌĆö valid, now rejected):**

```json
{
  "test_id": "plan_start_happy_path",
  "description": "plan_start creates plan.json with correct structure",
  "precondition": {
    "state_files": {
      ".nexus/state/plan.json": null
    }
  },
  "action": {
    "tool": "plan_start",
    "params": {
      "topic": "Introduce conformance test fixtures",
      "issues": ["What format?", "Which tools?"],
      "research_summary": "Reviewed existing state-schemas."
    }
  },
  "postcondition": {
    "return_value": {
      "$.created": true,
      "$.issueCount": 2
    }
  }
}
```

**After (v0.4.0 fixture ŌĆö valid):**

```json
{
  "test_id": "plan_start_happy_path",
  "description": "plan_start creates plan.json with correct structure",
  "covers": {
    "state_schemas": {
      "plan.schema.json": ["id", "topic", "issues[].id", "issues[].title", "issues[].status", "research_summary", "created_at"]
    },
    "return_value": {
      "plan_start": ["created", "plan_id", "topic", "issueCount"]
    }
  },
  "precondition": {
    "state_files": {
      ".nexus/state/plan.json": null
    }
  },
  "action": {
    "tool": "plan_start",
    "params": {
      "topic": "Introduce conformance test fixtures",
      "issues": ["What format?", "Which tools?"],
      "research_summary": "Reviewed existing state-schemas."
    }
  },
  "postcondition": {
    "return_value": {
      "$.created": true,
      "$.issueCount": 2
    }
  }
}
```

The `covers.state_schemas` object maps a schema filename to an array of field paths (using dot-notation and `[]` for array items). The `covers.return_value` object maps a tool name to an array of return-value field names.

### 2. Declare routing-only parameters in `uncovered_params`

Some `action.params` fields are routing keys that select a code path rather than representing data stored in state or returned to the caller. These cannot be covered by `covers` assertions because they do not appear as schema fields. Declare them in `uncovered_params` so the validator does not flag them as missing coverage.

**Before (v0.3.x ŌĆö valid, now the validator will flag `action` and `issue_id` as uncovered):**

```json
{
  "test_id": "plan_update_remove_happy_path",
  "action": {
    "tool": "plan_update",
    "params": {
      "action": "remove",
      "issue_id": 1
    }
  }
}
```

**After (v0.4.0 ŌĆö valid, routing params declared):**

```json
{
  "test_id": "plan_update_remove_happy_path",
  "covers": {
    "state_schemas": {
      "plan.schema.json": ["issues[].id"]
    },
    "return_value": {
      "plan_update": ["removed", "issue.id"]
    }
  },
  "uncovered_params": ["action", "issue_id"],
  "action": {
    "tool": "plan_update",
    "params": {
      "action": "remove",
      "issue_id": 1
    }
  }
}
```

Common routing-only parameters by tool:

| Tool | Routing-only params |
|---|---|
| `plan_update` | `action`, `issue_id` (for remove/edit/reopen) |
| `task_update` | `task_id`, `action` |
| `task_close` | `task_id` |

### 3. Integrate `validate:conformance` into CI

Add the validator as a required step in your CI pipeline before any release or publish step.

**Example (GitHub Actions):**

```yaml
- name: Validate conformance coverage
  run: bun run validate:conformance
```

The command exits with code 0 on full coverage and code 1 with a diagnostic report on any gap. A non-zero exit blocks the pipeline.

The validator checks:
- Every field in each `state-schema` is covered by at least one fixture's `covers.state_schemas` entry.
- Every `action.params` key is either named in `covers` assertions or declared in `uncovered_params`.
- No fixture has both `action` and `event` at the top level (oneOf enforcement).

---

### 4. Adopt event-based fixtures (optional but recommended)

v0.4.0 introduces `conformance/lifecycle/` with 5 event fixtures: `session-start.json`, `session-end.json`, `agent-spawn.json`, `agent-complete.json`, `agent-resume.json`.

Event fixtures use an `event` key instead of `action`:

```json
{
  "test_id": "session_start_creates_runtime",
  "description": "When a session starts, the harness writes runtime.json with a session_id and started_at timestamp",
  "covers": {
    "state_schemas": {
      "runtime.schema.json": ["session_id", "started_at"]
    }
  },
  "event": {
    "type": "session_start"
  },
  "postcondition": {
    "state_files": {
      ".nexus/state/runtime.json": {
        "$.session_id": { "type": "string", "minLength": 1 },
        "$.started_at": { "type": "iso8601" }
      }
    }
  }
}
```

If you maintain consumer-local lifecycle fixtures, adopt the `event` top-level key and remove any `action` key. Mixed fixtures (both `action` and `event`) are rejected by the v0.4.0 schema.

## Behavior Changes

- `fixture.schema.json` now rejects any fixture object that omits `covers`. Previously, `covers` was not present in the schema at all.
- The `oneOf` branch at the top level enforces mutual exclusion of `action` and `event`. A fixture that previously had an `action` key alongside an experimental `event` key will fail validation.
- `uncovered_params` is a new optional string-array field. If a fixture's `action.params` contains keys not named in `covers` assertions and not listed in `uncovered_params`, the validator reports a gap.

## Step-by-Step Migration Checklist

- [ ] **1.** Update your `package.json` dependency pin from `^0.3.x` to `^0.4.0`.
- [ ] **2.** Install v0.4.0.
- [ ] **3.** Run `bun run validate:conformance`. Note all reported gaps.
- [ ] **4.** Add `covers` to every custom fixture that is missing it. Use the before/after examples above as a template.
- [ ] **5.** Add `uncovered_params` to any fixture whose `action.params` contain routing-only keys.
- [ ] **6.** If any custom fixture has both `action` and `event` at the top level, remove the `event` key (or split into a separate fixture file).
- [ ] **7.** Re-run `bun run validate:conformance`. Confirm exit code 0.
- [ ] **8.** Add `bun run validate:conformance` to your CI pipeline.
- [ ] **9.** Optionally, add lifecycle fixtures for `session_start`, `session_end`, `agent_spawn`, `agent_complete`, `agent_resume` from `conformance/lifecycle/` as a reference.
- [ ] **10.** Run your consumer repo's full test suite to verify the migration.
