**Release date**: 2026-04-17
**Type**: Minor release (additive-with-obligation — Hook Event Runtime Mapping appendix 작성 의무 1건)
**Affected consumers**: claude-nexus, opencode-nexus, codex-nexus
**Tracking issues**: [GH #21](https://github.com/moreih29/nexus-core/issues/21), [GH #22](https://github.com/moreih29/nexus-core/issues/22), [GH #23](https://github.com/moreih29/nexus-core/issues/23)

## Summary

v0.11.0은 Plan session #7의 §9(lifecycle hook event catalogue) 거버넌스 재정립 결과를 실행한다. §9 전체를 harness-agnostic prose로 재작성하고, RFC 2119 conformance tag를 도입하며(MUST 5 bullet + SHOULD ~19 bullet + MAY ~6 bullet), consumer-owned Hook Event Runtime Mapping을 canonical appendix 의무로 확립한다. 3 active consumer(codex-nexus 신규 편입) 상태가 nexus-core 전반에 반영된다.

변경 surface는 5개 Issue로 구성된다. Issue 1(RFC 2119 hybrid, Option C): MUST dual gate 적용 — conformance/state-schemas/* 또는 conformance/lifecycle/*로 assertable이면서 3-of-3 consumer 필요성 empirical 인정 항목만 MUST 승격. Issue 2(A+2 합성안): event 이름 유지, consumer-owned `harness-content/nexus-hook-mapping.md`로 매핑 분리, v0.8.0 `harness_docs_refs` 선례 재사용. Issue 3(narrow 판정): DROP 9 bullet은 0-of-3 empirical 또는 harness-specific — non-breaking 제거. Issue 4(공통 패턴 5건): P1/P2/P5/P7 §9 prose 조정, P3 preamble 신설. Issue 5(3 active consumer 반영): `ecosystem.md`, `boundaries.md`, `README.md`, `CONSUMING.md`, `evolution.md` 드리프트 수정.

파괴적 변경 surface는 단 1건: 각 consumer의 `harness-content/nexus-hook-mapping.md` 신설 의무. MUST 5 bullet은 3-of-3 already-implemented이며, DROP 9 bullet은 0-of-3 not-implemented이므로 consumer code 변경을 유발하지 않는다.

## Breaking Change Surface

### (1) Mapping appendix 작성 의무 — `harness-content/nexus-hook-mapping.md` 신설

nexus-core §9 Appendix "Hook Event Runtime Mapping (consumer-owned)"이 신설됨. 각 consumer는 자기 repo에 `harness-content/nexus-hook-mapping.md`를 작성하여 §9가 정의하는 8 conceptual event(`session_start` / `user_message` / `subagent_spawn` / `subagent_complete` / `pre_tool_use` / `post_tool_use` / `session_end` / `context_compact`)가 자기 harness의 runtime hook surface에 어떻게 매핑되는지 문서화해야 한다.

파일을 신설하지 않으면 `skills/nx-run/meta.yml`의 `harness_docs_refs: nexus_hook_mapping` 토큰이 미해석 상태로 남아 `context_compact` hook 의존 skill context가 주입되지 않는다.

### (2) `nexus_hook_mapping` 토큰 수용 의무 — skill expander 처리 추가

`skills/nx-run/meta.yml`에 `harness_docs_refs: nexus_hook_mapping` 항목이 추가됨. Consumer의 skill build pipeline은 이 토큰을 인식하여 `harness-content/nexus-hook-mapping.md` 내용을 skill context로 주입하도록 expander 로직을 구현해야 한다. 토큰 schema는 enum 제약 없는 open string array(v0.8.0 `harness_docs_refs` 선례와 동일) — 기존 expander가 unknown token을 무시 처리하면 기능상 무해하지만, `nexus_hook_mapping` 미처리 시 mapping context 미주입으로 skill 동작 저하가 발생한다.

### 비파괴적 변경 (참고)

- **§9 DROP 9 bullet**: 0-of-3 empirical(아무 consumer도 구현하지 않음) + harness-specific guidance — normative prose에서 제거되나 consumer code 변경을 유발하지 않는다. 관련 구현이 있다면 제거를 권고하나 강제하지 않는다.
- **MUST 5 bullet 도입**: 3-of-3 already-implemented — consumer code 변경 없음. conformance assertion 경로만 명시화됨.
- **`conformance/lifecycle/session-end.json` 신설**: 새 fixture는 assertion coverage 확장. consumer conformance suite에 포함을 권장하나 강제하지 않는다.

## Classification Framework

v0.11.0에서 확립된 판단 기준. 향후 §9 업스트림 제안 수용/거부 시 동일 기준 적용.

### MUST dual gate

§9 bullet이 MUST로 승격되려면 두 조건을 모두 충족해야 한다.

**(A) Conformance-assertable**: `conformance/state-schemas/*` 또는 `conformance/lifecycle/*`로 기계적으로 assert 가능한 항목이어야 한다. prose-only guidance는 SHOULD 이하.

**(B) 3-of-3 empirical**: 3개 active consumer 모두가 해당 동작을 이미 구현하고 있음이 empirical로 확인되어야 한다. 2-of-3 이하는 SHOULD; 1-of-3 이하는 MAY.

v0.11.0 MUST 5 bullet은 이 dual gate를 통과한 항목이다.

### Negative MUST 범주

삭제 또는 보존 불변식을 표현하는 MUST. assertion은 `postcondition.state_files[path] != null` 패턴으로 수행. `conformance/lifecycle/session-end.json`의 "삭제 금지 파일 목록" fixture가 첫 번째 사례.

### Narrow 판정 기준

공통 intent 판별 기준: hook timing + target surface + data shape 세 축이 모두 일치해야 intent 공통으로 판단한다. 세 축 중 하나라도 불일치하면 narrow — §9 bullet에 포함하지 않는다.

Broad(purpose-matching) 기준은 §9를 hook-level 계약이 아닌 capability wish-list로 전락시키므로 기각한다. DROP 9 bullet이 narrow 판정 결과다.

### Evidence threshold (boundaries.md §Canonical specifics)

| RFC 2119 | Consumer evidence | 비고 |
|----------|------------------|------|
| MUST | 3-of-3 ∧ conformance-assertable | dual gate |
| SHOULD | 2-of-3 | |
| MAY | 1-of-3 | |
| (omit / DROP) | 0-of-3 | narrow 판정 또는 harness-specific |

## Consumer Action Checklist

### claude-nexus

1. `package.json`의 `@moreih29/nexus-core` devDependency를 `^0.11.0`으로 업데이트.

2. `bun run dev`(또는 consumer 빌드 명령)으로 재빌드. `verifyBodyHash` gate가 `skills/nx-run` 새 body hash를 수용.

3. **`harness-content/nexus-hook-mapping.md` 신설** — Claude Code hook API 8종을 §9 8 conceptual event에 매핑하여 문서화.
   - `SessionStart` → `session_start`
   - `UserPromptSubmit` → `user_message`
   - `SubagentStart` → `subagent_spawn`
   - `SubagentStop` → `subagent_complete`
   - `PreToolUse` → `pre_tool_use`
   - `PostToolUse` → `post_tool_use`
   - `Stop` → `session_end`
   - `PreCompact` 또는 `PostCompact` → `context_compact` (harness에서 노출하는 hook 명시)
   - 각 매핑 행에 "hook name", "fires when", "§9 event", "notes" 컬럼 포함 권장.

4. **skill expander `nexus_hook_mapping` 토큰 구현** — `harness_docs_refs` 처리 로직에서 `nexus_hook_mapping` 토큰을 인식하여 위 파일 내용을 skill context로 주입. v0.8.0에서 도입한 `harness_docs_refs` expander 패턴 그대로 적용.

5. **DROP §9 bullet 대응** — DROP된 9 bullet은 구현 여부 0-of-3 empirical이므로 consumer code에 해당 로직이 없는 경우 추가 작업 없음. 만약 `session_start`의 stale-state 경고, `user_message`의 `inline_action` handler 호출, `subagent_spawn`의 `task_id` 기록 등 관련 구현이 존재한다면 §9 normative 범위 밖임을 인지하고 제거 또는 consumer-local 정책으로 이관.

6. **MUST 5 bullet 확인** — 3-of-3 already-implemented이므로 신규 구현 불필요. 그러나 `conformance/lifecycle/session-end.json` fixture가 추가된 만큼 conformance 실행 시 새 fixture가 포함됨을 확인.

### opencode-nexus

1. `package.json`의 `@moreih29/nexus-core` devDependency를 `^0.11.0`으로 업데이트 후 빌드 재실행.

2. **`harness-content/nexus-hook-mapping.md` 신설** — OpenCode hook API를 §9 8 conceptual event에 매핑하여 문서화.
   - `event(session.created)` → `session_start`
   - `chat.message` → `user_message` (주: `experimental.chat.system.transform`이 user_message의 일부를 처리하는 경우 분산 매핑을 명시)
   - `tool.execute.before` → `subagent_spawn` 및/또는 `pre_tool_use` (OpenCode에서 두 event가 동일 hook에 collapse되는 경우 해당 collapse 패턴을 "collapse note"로 명시)
   - `tool.execute.after` → `subagent_complete` 및/또는 `post_tool_use` (동일하게 collapse 패턴 명시)
   - `event(session.deleted)` → `session_end`
   - `experimental.session.compacting` → `context_compact` (experimental status를 note로 기재)
   - collapse 및 분산 매핑 패턴을 각 행의 "notes" 컬럼에 명시하여 ambiguity 제거.

3. **skill expander `nexus_hook_mapping` 토큰 구현** — `harness_docs_refs` 처리 로직에서 `nexus_hook_mapping` 토큰을 인식하여 위 파일 내용을 skill context로 주입. OpenCode harness의 skill build 파이프라인 적용 방식을 따라 구현.

4. **DROP §9 bullet 대응** — DROP된 9 bullet 관련 구현이 OpenCode harness에 존재한다면 consumer-local 정책으로 이관 또는 제거 검토. §9 normative 범위에서 제외됨.

5. **MUST 5 bullet 확인** — 3-of-3 already-implemented. `conformance/lifecycle/session-end.json` 신규 fixture가 conformance suite에 포함됨을 확인.

### codex-nexus

1. `package.json`의 `@moreih29/nexus-core` devDependency를 `^0.11.0`으로 업데이트 후 빌드 재실행.

2. **`harness-content/nexus-hook-mapping.md` 신설** — Codex hook API를 §9 8 conceptual event에 매핑하여 문서화.
   - `SessionStart` → `session_start` 및 `subagent_spawn` (Codex에서 session_start와 subagent_spawn이 동일 hook에 collapse되는 경우 collapse 패턴을 "collapse note"로 명시)
   - `UserPromptSubmit` → `user_message`
   - `PreToolUse` → `pre_tool_use`
   - `PostToolUse` → `post_tool_use`
   - `Stop` → `session_end` 및 `subagent_complete` (Stop이 두 event를 cover하는 경우 collapse 패턴 명시)
   - `context_compact` → Codex runtime이 compact hook을 노출하지 않는 경우 "not exposed — transcript-based detection only" 또는 해당 hook 부재를 명시
   - transcript-based subagent detection 방식을 사용하는 경우 detection 기법을 notes에 서술.

3. **`harness_docs_refs` expander 구현** — codex-nexus는 신규 consumer이므로 `harness_docs_refs` expander가 미구현 상태일 수 있음. v0.8.0 선례 토큰 처리 로직을 참조하여 expander를 신규 구현하고, `nexus_hook_mapping` 토큰을 우선 처리 대상으로 등록.

4. **DROP §9 bullet 대응** — codex-nexus는 신규 편입 consumer이므로 DROP된 9 bullet을 구현한 경우가 없을 가능성이 높음. 기존 코드베이스에 해당 패턴이 존재하지 않으면 추가 작업 없음.

5. **MUST 5 bullet + conformance suite 초기 설정** — 3-of-3 already-implemented로 분류된 MUST 5 bullet이 codex-nexus에서도 충족되는지 점검. `conformance/lifecycle/session-end.json`을 포함한 conformance suite를 빌드 파이프라인에 통합.

> **codex-nexus 신규 편입 특별 사항**: codex-nexus는 v0.11.0 기준으로 nexus-core 3번째 active consumer로 공식 반영된다(`ecosystem.md`, `README.md`, `CONSUMING.md` 업데이트). 위 5개 항목 외에 `CONSUMING.md §File Contracts` 테이블의 모든 파일 계약을 확인하고 consumer-local 구현을 정렬할 것을 권장한다.

## Rejected Items Reference

이 항목들은 Plan session #7에서 nexus-core canonical에 포함하지 않기로 결정되었다. 각 거부 근거가 명시된다.

### Issue 1 — RFC 2119 방식 선택지 거부

**Option A (strictly normative) 기각**: 집행 불가 MUST를 양산한다. 3 consumer가 동시에 violating 상태가 되며, evolution.md §완화 근거 및 현재 0-of-3 empirical 항목에 MUST를 부여하는 것은 conformance schema와 모순된다.

**Option B (reference only — RFC 2119 미사용) 기각**: conformance state-schemas와 prose 일관성을 파괴한다. §9의 MUST 5 bullet은 이미 canonical schema로 assert되므로 RFC 2119 없이 prose만 남기면 normative 계층 구조가 붕괴된다.

**Option D (split docs — physical separation) 기각**: premature structure. 1인 유지 환경에서 두 파일 간 드리프트 소스가 증가하며, 현재 §9 규모는 분리를 정당화하지 않는다.

### Issue 2 — hook event 재명명 또는 전면 재작성 거부

**full event-based §9 rewrite (Option B — 8 conceptual hook frame 전면 채택) 기각**:
- R1: 기존 §9 rewrite 비용 대비 marginal gain이 불충분하다.
- R2: "8 conceptual hook" frame 자체가 단일 consumer(Claude Code) 유래다 — 3 consumer에 generalizable하다는 evidence가 아직 확립되지 않았다.
- R3: "conceptual" note가 normative로 오인될 위험이 있다.
- A+2 합성안(event 이름 유지 + consumer-owned mapping)이 비용 효율적으로 동일 목표를 달성한다.

### Issue 3 — DROP 9 bullet 관련 거부

아래 항목들은 **narrow 판정** 기준(hook timing + target surface + data shape 세 축 불일치) 또는 0-of-3 empirical로 §9 normative prose에서 제거됨. consumer-local 필요가 있다면 harness-local 정책으로 유지 가능.

- `session_start`: stale-state warning bullet — harness-specific UX concern
- `session_start`: knowledge index load bullet — harness-specific indexing mechanism
- `user_message`: `inline_action` handler call bullet — 0-of-3 empirical, harness-specific dispatch
- `user_message`: knowledge file counts bullet — 0-of-3 empirical, harness-specific observability
- `subagent_spawn`: structured task context bullet — data shape 불일치 (3 consumer shape 상이)
- `subagent_spawn`: `task_id` field mention — schema에 없는 필드, 3-of-3 기록 evidence 불충분
- `subagent_complete`: acceptance reminder bullet — dispatcher policy, not hook semantics
- `subagent_complete`: `task_update` auto call bullet — harness-specific tool invocation
- `session_end`: active plan warning bullet — harness-specific plan state management

### Issue 4 — 공통 패턴 watchlist 항목 (보류)

**A1 `tool-log.jsonl` canonical 승격 기각**: 3 consumer의 실제 필드 shape가 상이하다(opencode / claude / codex 각자 독립 schema). canonical 확정은 premature. evolution.md §90일 재평가 지표 item 10으로 watchlist 등록 — 3-of-3 adoption evidence 확인 후 재검토.

**A3 `agent-tracker` `task_id` field schema 추가 기각**: `docs/consumer-implementation-guide.md` §9가 `task_id`를 언급했으나 `conformance/state-schemas/agent-tracker.schema.json`에 해당 field가 없다. 3 consumer의 실제 기록 여부 evidence 불충분. schema 확장은 premature — guide에서 `task_id` 언급을 제거하는 방향이 올바르다.
