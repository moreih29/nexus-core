# Migration Guide: v0.6.x → v0.7.0

**Release date**: 2026-04-14
**Type**: Minor correctness release (namespace isolation + schema tightening)
**Affected consumers**: claude-nexus, opencode-nexus, nexus-code
**Tracking issue**: [GH #16](https://github.com/moreih29/nexus-core/issues/16)

---

## Overview

v0.7.0 ships a focused set of correctness changes to the `agent-tracker.json` contract. This is a pre-v1 minor correctness release — it tightens an existing contract rather than removing one, and does not include a cooling period commitment against future breaking changes.

Three changes are breaking for active consumers:

1. `agent-tracker.json` moves from the shared state root (`.nexus/state/`) to a per-harness namespace (`.nexus/state/{harness-id}/`), eliminating silent clobber between co-running harnesses.
2. The schema's `required` array is reduced from 6 fields to 2 (`harness_id`, `started_at`). The remaining 4 fields (`agent_name`, `agent_id`, `status`, `resume_count`) are demoted to optional.
3. Lifecycle fixture placeholder token conventions (`{STATE_ROOT}`, `{HARNESS_ID}`) are introduced for conformance runners.

Accompanying documentation changes (`docs/nexus-outputs-contract.md §Shared filename convention`, `.nexus/context/ecosystem.md §Co-run scenarios`) are additive and require no consumer action beyond awareness.

**opencode-nexus** (Phase 1, active) must apply all checklist items. **claude-nexus** and **nexus-code** (Phase 2, awaiting) should apply the checklist before beginning active consumption.

## What changed

### `agent-tracker.json` path namespace isolation

- **Before (v0.6.0)**: `.nexus/state/agent-tracker.json` — a single file at the state root, shared across any harness writing to `.nexus/state/`.
- **After (v0.7.0)**: `.nexus/state/{harness-id}/agent-tracker.json` — each harness writes under its own subdirectory, e.g. `.nexus/state/claude-nexus/agent-tracker.json`.

This change prevents silent clobber when two harnesses run against the same workspace. The glob pattern `.nexus/state/*/agent-tracker.json` is the recommended cross-harness aggregation model for supervision tooling.

### Schema `required` field reduction (6 → 2)

The `conformance/state-schemas/agent-tracker.schema.json` required array is reduced:

| Field | v0.6.0 | v0.7.0 |
|---|---|---|
| `harness_id` | required | required |
| `started_at` | required | required |
| `agent_name` | required | optional |
| `agent_id` | required | optional |
| `status` | required | optional |
| `resume_count` | required | optional |

Consumers MUST assert only `harness_id` and `started_at` at write time. Optional fields MAY be written at consumer discretion.

### `agent_id` description revised to harness-specific opaque string

`agent_id` is now described in the schema as a harness-specific opaque identifier. Consumers MUST treat any `agent_id` value as opaque within its harness of origin. Parsing or inferring the structure of another harness's `agent_id` values is prohibited — cross-harness `agent_id` semantics are undefined.

### Fixture placeholder token convention introduced

`conformance/lifecycle/agent-spawn.json`, `agent-complete.json`, and `agent-resume.json` (the three lifecycle fixtures) now use placeholder tokens in path values:

- `{STATE_ROOT}` — resolves to `.nexus/state` in the consumer's runner configuration
- `{HARNESS_ID}` — resolves to the harness identifier from `event.params.harness_id` or equivalent runner config

This narrow-scope convention applies **only** to the three lifecycle fixtures listed above. All other conformance fixtures retain hardcoded paths. Consumers must implement token substitution in their conformance runner before running `validate:conformance`.

### `rule:harness-state-namespace` intent reframe

`.nexus/rules/neutral-principles.md` — the `harness-state-namespace` rule now frames its intent as **preventing silent schema divergence**, not merely prescribing directory structure. The practical effect: `agent-tracker.json` (a nexus-core-defined contract) is permitted under a harness namespace subdirectory while retaining the same filename. This is consistent with the `outputs-contract §Shared filename convention` introduced below.

### `docs/nexus-outputs-contract.md` — §Shared filename convention added

A new section defines the contract for files that appear under multiple harness namespaces with the same name. When nexus-core defines the schema and the filename, each harness MAY place its own instance under `.nexus/state/{harness-id}/{filename}` without ambiguity. `agent-tracker.json` is the first file governed by this convention.

### `.nexus/context/ecosystem.md` — §Co-run scenarios added

A new section documents the concept of multiple harnesses writing to the same `.nexus/state/` workspace simultaneously. This is orthogonal to the "flip" (switching between harnesses across sessions) scenario documented previously. Co-run is now first-class in ecosystem documentation. No consumer action is required; the section is informational.

## Consumer action checklist

Apply all items before upgrading to v0.7.0.

- [ ] Update the `AGENT_TRACKER_FILE` path constant (or equivalent) to `.nexus/state/{harness-id}/agent-tracker.json`, substituting your harness's literal identifier for `{harness-id}`.
- [ ] Add logic to your session-start hook to create the `.nexus/state/{harness-id}/` subdirectory if it does not exist, before writing `agent-tracker.json`.
- [ ] At `agent-tracker.json` write time, assert only the 2 required fields (`harness_id`, `started_at`). The optional fields (`agent_name`, `agent_id`, `status`, `resume_count`) MAY be written but MUST NOT be asserted as required.
- [ ] Treat `agent_id` values as harness-specific opaque strings. Do not parse or infer structure from another harness's `agent_id` values.
- [ ] Implement placeholder token substitution in your conformance runner: replace `{STATE_ROOT}` with `.nexus/state` and `{HARNESS_ID}` with your harness identifier (from `event.params.harness_id` or runner config) before evaluating lifecycle fixture path assertions.
- [ ] Re-read `rule:harness-state-namespace` in `.nexus/rules/neutral-principles.md` — confirm your understanding of the updated intent (silent schema divergence prevention, not directory-only prescription).
- [ ] If you are building supervision or aggregation tooling (nexus-code): use `.nexus/state/*/agent-tracker.json` glob to collect agent-tracker files across harnesses. Do not assume the file exists at the state root.

## Migration policy

`agent-tracker.json` is session-scoped; no user-initiated migration is required. Harnesses MAY silently remove any legacy `.nexus/state/agent-tracker.json` (root-level) at session start. Session-end cleanup already guarantees the file does not persist across sessions.

For installations where a previous session left a root-level `agent-tracker.json` in place, the recommended handling is described in `docs/consumer-implementation-guide.md` §Session start: check for the legacy path at session start and remove it if found. This is a MAY-level recommendation only — the session-scoped nature of the file means it will not survive a clean session end under any harness version.

No data migration tooling is provided by nexus-core for this change.

---

## Validation

After applying the consumer action checklist, run:

```bash
bun install
bun run validate
bun run validate:conformance
```

Both commands must exit with code 0. `validate:conformance` will exercise the updated lifecycle fixtures; confirm that your runner's placeholder token substitution is in place before running it, or fixture path assertions will fail.

## Directory structure example

```
# Before (v0.6.0)
.nexus/state/
├── plan.json
├── tasks.json
├── agent-tracker.json
├── edit-tracker.json
├── reopen-tracker.json
└── artifacts/

# After (v0.7.0)
.nexus/state/
├── plan.json
├── tasks.json
├── edit-tracker.json
├── reopen-tracker.json
├── artifacts/
├── claude-nexus/
│   └── agent-tracker.json
└── opencode-nexus/
    └── agent-tracker.json
```

Each harness maintains its own `agent-tracker.json` under its namespace. Shared state files (`plan.json`, `tasks.json`, `edit-tracker.json`, `reopen-tracker.json`, `artifacts/`) remain at the state root as before.
