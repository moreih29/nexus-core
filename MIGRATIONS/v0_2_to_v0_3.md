# Migration Guide: v0.2.x → v0.3.0

**Release date**: 2026-04-12
**Type**: Breaking change (`skills/nx-setup/` removed)
**Affected consumers**: claude-nexus, opencode-nexus, nexus-code

---

## Summary

v0.3.0 removes `skills/nx-setup/` from nexus-core. The `nx-setup` body.md was entirely Claude Code–specific (configuration paths, settings schema, plugin system, hook scripts), violating the harness-neutral principle. Consumers must now implement their own setup skill.

Additionally, all remaining harness-specific tool name references in body.md files have been neutralized, and the nx-init skill's `CLAUDE.md` hardcoding has been replaced with the abstract "instruction file" term.

---

## Breaking Change: `skills/nx-setup/` removed

### What changed

The `skills/nx-setup/` directory (body.md + meta.yml) has been deleted. The `manifest.json` no longer lists `nx-setup` as a skill.

### Why

Every step in nx-setup's body.md referenced Claude Code internals:
- `~/.claude/CLAUDE.md`, `~/.claude/settings.json`, `.claude/settings.local.json`
- `statusLine` JSON field + `~/.claude/hooks/nexus-statusline.sh`
- `enabledPlugins` + `context7@claude-plugins-official`

Abstracting this into a harness-neutral skeleton would produce an empty shell — the implementation details are inherently harness-specific.

### Consumer action required

1. **Remove `nx-setup` references** from your skill-loader, manifest-resolution, and bootstrap logic.
2. **Implement a local setup skill** in your consumer repo. See `CONSUMING.md` → "Setup Skill Contract" for the contract:
   - Scope selection (user-level vs project-level)
   - Status display configuration
   - Plugins/extensions installation
   - Knowledge initialization (nx-init integration)
3. **Update your dependency pin** from `^0.2.x` to `^0.3.0` after the local setup skill is validated.

The former `skills/nx-setup/body.md` (available in the v0.2.0 tag) can serve as a reference implementation for Claude Code consumers.

## Non-Breaking Changes

### body.md tool name neutralization

All harness-specific tool names in body.md files have been replaced with neutral capability expressions:

| Before | After |
|---|---|
| `Edit tool` | harness's file-editing primitive |
| `Write tool` | harness's file-creation primitive |
| `Read tool` | harness's file-reading primitive |
| `Bash tool` | shell command execution |
| `Agent tool` | harness's subagent spawn primitive |

Affected files: `skills/nx-init/body.md`, `skills/nx-sync/body.md`, `skills/nx-plan/body.md`.

### nx-init instruction file abstraction

`skills/nx-init/body.md` no longer hardcodes `CLAUDE.md` as the instruction file target. All action-context references now use the abstract term "instruction file". `meta.yml` gains `harness_docs_refs: ["instruction_file"]` so consumers can inject their harness-specific file name.

### Conformance fixture expansion

Two new standalone fixtures added to `conformance/tools/`:
- `task-update.json` — task status transition assertions
- `task-list.json` — task listing and `ready` set computation assertions

Tool coverage: 4/11 → 6/11.

## Step-by-Step Migration Checklist

- [ ] **1.** Search your codebase for `nx-setup` references. Remove or replace each one.
- [ ] **2.** Implement a local setup skill per the contract in `CONSUMING.md`.
- [ ] **3.** If your consumer uses `harness_docs_refs`, add an `instruction_file` entry to your harness docs directory mapping the abstract term to your harness's instruction file name.
- [ ] **4.** Run all `conformance/tools/*.json` fixtures (including the two new ones). All must pass.
- [ ] **5.** Update your dependency pin to `^0.3.0`.
- [ ] **6.** Run your consumer repo's test suite to verify the migration.
