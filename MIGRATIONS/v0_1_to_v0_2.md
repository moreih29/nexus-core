# Migration Guide: v0.1.x → v0.2.0

**Release date**: 2026-04-12
**Type**: Breaking change (vocabulary redesign — `schema_contract_version` 1.0 → 2.0)
**Affected consumers**: claude-nexus, opencode-nexus, nexus-code

---

## Summary

v0.2.0 redesigns `vocabulary/capabilities.yml` to be fully harness-neutral. The `harness_mapping` field — which previously listed concrete tool names per harness — is removed entirely. Consumers must now maintain a local `capability-map.yml` (or equivalent) that maps nexus-core's semantic capability descriptions to their own harness-specific tool names.

Additionally, two new optional fields are added to `skill.schema.json`, new validation gates are introduced, and new `conformance/` and `docs/` directories are published to npm.

---

## Breaking Change 1: `vocabulary/capabilities.yml` — `harness_mapping` removed

### What changed

The `harness_mapping` field is removed from every capability entry in `vocabulary/capabilities.yml`. The `schema_contract_version` field advances from `1.0` to `2.0`.

**Before (v0.1.x):**
```yaml
schema_contract_version: "1.0"
capabilities:
  - id: no_file_edit
    description: "Cannot create or modify files"
    harness_mapping:
      claude-nexus: ["Edit", "Write"]
      opencode-nexus: ["edit", "write"]
```

**After (v0.2.0):**
```yaml
schema_contract_version: "2.0"
capabilities:
  - id: no_file_edit
    intent: "Agent must not create or modify files on the filesystem"
    blocks_semantic_classes: ["file_write", "file_create", "file_overwrite"]
    prose_guidance: "Do not use any tool or capability that writes, creates, or overwrites files. Read-only file access is permitted."
```

### New fields

| Field | Type | Purpose |
|---|---|---|
| `intent` | string | Normative statement of what the capability restricts |
| `blocks_semantic_classes` | string[] | Abstract semantic class labels consumers map to concrete tools |
| `prose_guidance` | string | Human/LLM-readable instruction suitable for prompt injection |

### New capability: `no_shell_exec`

A fourth capability entry, `no_shell_exec`, is added using the X3 schema (intent + blocks_semantic_classes + prose_guidance). No canonical agent in nexus-core references it at v0.2.0 — it is vocabulary-only, available for consumer agents to opt in to.

### Consumer action required

**Step A — Remove `harness_mapping` reads.** Audit your harness for any code that reads `capabilities[*].harness_mapping[yourHarnessName]` to resolve tool names. That path no longer exists in v0.2.0.

**Step B — Create a local `capability-map.yml`.** Add a file to your consumer repo that maps each `blocks_semantic_class` label to your harness's concrete tool names. This file is consumer-local and is never shipped by nexus-core.

Example structure:
```yaml
# capability-map.yml — consumer-local, not provided by nexus-core
semantic_class_map:
  file_write:
    - YourWriteTool
  file_create:
    - YourWriteTool
  file_overwrite:
    - YourWriteTool
    - YourEditTool
  shell_exec:
    - YourShellTool
```

**Step C — Read `prose_guidance` for prompt injection.** The `prose_guidance` value in `manifest.json` is a direct replacement for the narrative you previously assembled from `harness_mapping` tool lists.

**Step D — Add a CI assertion.** Add a test that verifies your `capability-map.yml` covers every `blocks_semantic_class` present in the installed `vocabulary/capabilities.yml`. This prevents silent gaps when nexus-core adds new semantic classes in future releases.

**Step E — Update `schema_contract_version` checks.** Update any consumer code that asserts `schema_contract_version === "1.0"` to accept `"2.0"`.

## Breaking Change 2: `body_hash` values changed for all agents and skills

### What changed

All 9 agent `body.md` files and all 5 skill `body.md` files were rewritten to remove harness-specific tool name references. Any concrete tool name that appeared in body content has been replaced with neutral capability phrasing.

Because the body content changed, all `body_hash` values in `manifest.json` are different in v0.2.0.

### Consumer action required

If your consumer code uses `verifyBodyHash` to detect tampering or verify file integrity, you must not rely on hardcoded hashes from v0.1.x.

**Correct approach**: Read expected hashes from the installed `manifest.json` at runtime. The `manifest.json` shipped with each version is the authoritative source of hashes for that version.

```
# Pseudocode
manifest = readJson("node_modules/@moreih29/nexus-core/manifest.json")
expected = manifest.agents[agentId].body_hash
ok = verifyBodyHash(rawBodyContent, expected)
```

See [`.nexus/memory/consumer-lib-reference.md`](./../.nexus/memory/consumer-lib-reference.md) for the `verifyBodyHash` pseudocode reference.

## Non-Breaking Addition 1: New optional fields in `skill.schema.json`

These fields are optional. Existing `meta.yml` files that omit them remain schema-valid.

| Field | Type | Constraints | Purpose |
|---|---|---|---|
| `summary` | string | optional, 10–120 chars | Short one-liner for UI and catalog rendering |
| `harness_docs_refs` | string[] | optional | References to consumer-local harness-specific documentation |
| `manual_only` | (existing) | now has normative description | Normative language clarified; behavior unchanged |

No consumer action is required for compatibility. Consumers who want to surface `summary` in a UI catalog or link harness docs via `harness_docs_refs` may begin populating those fields in their local skill overrides.

## Non-Breaking Addition 2: `conformance/` directory

A new `conformance/` directory is published to npm in v0.2.0.

| Path | Contents |
|---|---|
| `conformance/state-schemas/*.json` | JSON Schemas for plan, tasks, history, runtime, and agent-tracker state files |
| `conformance/tools/*.json` | Tool behavioral conformance fixtures (plan_start, plan_decide, task_add, task_close) |
| `conformance/scenarios/*.json` | Lifecycle scenario fixtures (full-plan-cycle, task-deps-ordering) |
| `conformance/schema/fixture.schema.json` | JSON Schema for the conformance fixture format |

### Consumer action (recommended)

Run the tool conformance fixtures against your harness implementation to verify compatibility. The fixture format is described by `conformance/schema/fixture.schema.json`.

```
# Pseudocode
fixtures = readDir("node_modules/@moreih29/nexus-core/conformance/tools/")
for fixture in fixtures:
  assertConformance(yourImplementation, fixture)
```

This step is recommended but not required for v0.2.0 compatibility. Future major releases may treat non-conforming behavior as breaking.

## Non-Breaking Addition 3: `docs/` directory

A new `docs/` directory is published to npm in v0.2.0.

| File | Contents |
|---|---|
| `docs/nexus-tools-contract.md` | Semantic specifications for all 11 Nexus-core tools |
| `docs/nexus-state-overview.md` | State file lifecycle and tool interaction overview |
| `docs/nexus-layout.md` | Canonical `.nexus/` directory structure |
| `docs/behavioral-contracts.md` | Behavioral contracts: state machines, resume tiers, permissions, `manual_only`, NL trigger boundary |

No consumer action is required. These documents replace inline documentation previously scattered across body.md files.

## Non-Breaking Addition 4: New validation gates (nexus-core contributors only)

These changes affect the `bun run validate` pipeline inside nexus-core and do not affect consumers reading the published npm package.

| Gate | Description |
|---|---|
| G11 (`G11-tag-trigger`) | New: validates that every `trigger` value in `vocabulary/tags.yml` corresponds to a defined tag `id` |
| G5' (capability integrity) | Redefined: validates `intent`, `blocks_semantic_classes`, and `prose_guidance` in place of the old `harness_mapping` structure |
| G6 (harness lint) | Expanded: now scans `agents/**/body.md` and `skills/**/body.md` in addition to previously covered paths |

## Step-by-Step Migration Checklist

Complete these steps in order after bumping your `@moreih29/nexus-core` dependency to `^0.2.0`:

- [ ] **1.** Install v0.2.0 (`npm install` or equivalent).
- [ ] **2.** Search your codebase for reads of `capabilities[*].harness_mapping`. Remove or replace each one.
- [ ] **3.** Create a local `capability-map.yml` mapping each `blocks_semantic_class` to your harness tool names.
- [ ] **4.** Update any `schema_contract_version` assertions from `"1.0"` to `"2.0"`.
- [ ] **5.** Update any hardcoded `body_hash` values to read from `manifest.json` at runtime instead.
- [ ] **6.** Add a CI assertion that your `capability-map.yml` covers all `blocks_semantic_classes` in `vocabulary/capabilities.yml`.
- [ ] **7.** (Recommended) Run `conformance/tools/*.json` fixtures against your tool implementations.
- [ ] **8.** Review `docs/behavioral-contracts.md` and `docs/nexus-tools-contract.md` for behavioral gaps.
- [ ] **9.** Run your consumer repo's test suite to verify the migration.
