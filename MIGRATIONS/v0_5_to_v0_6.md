# Migration Guide: v0.5.x → v0.6.0

**Release date**: 2026-04-14
**Type**: Single breaking change (`runtime.json` common schema hard removal)
**Affected consumers**: claude-nexus, opencode-nexus, nexus-code

---

## Summary

v0.6.0 ships one breaking change:

1. `runtime.json` is removed from nexus-core's common schema layer entirely. The schema file, its two lifecycle fixtures (`session-start.json`, `session-end.json`), the `session_start` and `session_end` event type enum values, and all documentation references are deleted.

Consumer action is required if your harness still contains write or delete logic for `runtime.json` under the common state path. See the checklist below.

nexus-code is in planning-only status and has not begun consuming nexus-core; no migration action is required for that harness.

---

## Breaking Change 1: `runtime.json` common schema hard removal

### What changed

The following artifacts are removed from nexus-core:

| Artifact | Change |
|---|---|
| `conformance/state-schemas/runtime.schema.json` | Deleted |
| `conformance/lifecycle/session-start.json` | Deleted |
| `conformance/lifecycle/session-end.json` | Deleted |
| `conformance/schema/fixture.schema.json` — `event.type` enum | `session_start`, `session_end` values removed |
| `conformance/lifecycle/agent-spawn.json` — `preconditions` | `runtime.json` precondition entry removed |
| `docs/nexus-outputs-contract.md` | `runtime.json` references removed |
| `docs/nexus-layout.md` | `runtime.json` entry removed from state layout |
| `docs/nexus-state-overview.md` | `runtime.json` section removed |
| `docs/consumer-implementation-guide.md` | `runtime.json` writer step removed |
| `docs/CONSUMING.md` | `runtime.json` references removed |
| `.nexus/rules/neutral-principles.md` (line 167) | `runtime.json` removed from common schema list |
| `conformance/README.md` | `runtime.json` row removed |
| `conformance/lifecycle/README.md` | `session-start`, `session-end` rows removed |

`package.json` version is bumped from `0.5.0` to `0.6.0`.

### Why

Across both active consumers (claude-nexus GH #14, opencode-nexus GH #15), every write site for `runtime.json` was found to be no-op or dead code — the file was written at session start and deleted at session end, but no consumer read-site consumed it. nexus-core was maintaining a contract (schema + fixtures + documentation) that reflected zero actual usage. Removing a false contract from the canonical source is preferable to leaving it in place as a maintenance burden that misleads future consumers.

### Consumer action required

**claude-nexus** (tracked in GH #14) and **opencode-nexus** (tracked in GH #15):

- [ ] Remove write logic in the session-start hook that creates `.nexus/state/runtime.json`.
- [ ] Remove delete logic in the session-end hook that removes `.nexus/state/runtime.json`.
- [ ] Remove any reference to `conformance/lifecycle/session-start.json` or `session-end.json` in your local conformance test runner configuration.
- [ ] If your CI asserts that `session_start` or `session_end` appear in `event.type` enum coverage, remove those assertions.
- [ ] If you extended `runtime.json` with harness-local fields and still need those fields at runtime, move them to a harness-local namespace (see Harness-local Extension below).

**nexus-code**: No action required — consuming has not started.

## Harness-local Extension (optional)

If your harness needs to persist runtime-like configuration that was previously stored in `runtime.json`, use the harness-local state extension pattern instead of the removed common schema.

### Pattern

Store the data under your harness-scoped namespace:

```
.nexus/state/{harness-id}/session-config.json
```

or, using the `.extension.json` convention:

```
.nexus/state/{harness-id}/runtime.extension.json
```

### Constraints

- **Do not reuse the filename `runtime.json`** under the harness-local namespace. `neutral-principles.md` prohibits reusing common schema filenames to avoid confusion with the (now deleted) common contract.
- Use a harness-specific name such as `session-config.json`, `session-meta.json`, or a similarly scoped identifier.
- Define your own JSON Schema for this file inside your harness repository (e.g. `state-schemas/session-config.schema.json`). nexus-core no longer provides or validates this schema.

### Example

```json
// .nexus/state/claude-nexus/session-config.json
{
  "harness_id": "claude-nexus",
  "harness_version": "1.2.0",
  "session_started_at": "2026-04-14T00:00:00Z"
}
```

See `docs/nexus-outputs-contract.md` §Harness-local State Extension for the full convention on namespacing and schema ownership.

## Validation

After applying the consumer action checklist, run:

```
bunx nexus-validate-conformance
```

The conformance suite no longer includes `session-start.json` or `session-end.json` fixtures, so there are no new fixture assertions to satisfy. A clean exit code 0 confirms your local test configuration does not reference deleted fixtures.

## Notes

- **nexus-code**: Consuming has not started (planning phase only). This removal has no impact.
- **claude-nexus, opencode-nexus**: GH #14 and #15 confirmed that `runtime.json` writes were already no-op. Hook logic removal is the only required step — no data migration is needed because `runtime.json` was write-only with no surviving read-site.
- **No rollback path**: Per nexus-core's Forward-only 완화 policy, once a common schema is removed at a semver major boundary it is not restored. If a use case emerges that requires runtime state, define it as a harness-local extension.
