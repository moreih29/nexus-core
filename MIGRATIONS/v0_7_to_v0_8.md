# Migration Guide: v0.7.x → v0.8.0

**Release date**: 2026-04-14
**Type**: Minor additive release (macro abstraction layer + invocation vocabulary)
**Affected consumers**: claude-nexus, opencode-nexus

---

## Summary

v0.8.0 ships the Spec γ macro abstraction layer and a new invocation vocabulary:

1. `body.md` 내 하네스 특화 tool 호출 13건이 `{{primitive_id key=val}}` 매크로 체계로 추상화됨 — harness-specific tool 이름이 body.md에서 완전히 제거됨.
2. `vocabulary/invocations.yml` 신설 — `skill_activation`, `subagent_spawn`, `task_register`, `user_question` 4개 semantic primitive 정의.
3. lint G6 (`body.md harness-specific tool name` 검사) 확장 및 `neutral-principles.md` rule 업데이트.
4. **Consumer action required**: 매크로 expander 구현 + `invocation-map.yml` 신설 + `harness-content/slash_command_display.md` 신설 + `generate` 재빌드 필수.
5. pre-v1 minor 릴리스 + nx-car marker 일관 (v0.2 / 0.4 / 0.5 / 0.6 / 0.7 선례).

## Macro Grammar Reference (Spec γ)

### 토큰 경계

```
{{primitive_id key=val key2="val with spaces"}}
```

- 열림: `{{`, 닫힘: `}}`
- `primitive_id`는 `vocabulary/invocations.yml`의 `id` 값과 1:1 대응.
- 파라미터는 `key=val` 형식 — 값에 공백 포함 시 `key="..."` 큰따옴표 사용.
- 배열: `key=[item1, item2]`, 오브젝트 인라인: `key={k: v}`.

### Heredoc (multiline value)

Ruby-style heredoc으로 긴 프롬프트 값을 표현한다:

```
{{subagent_spawn target_role=writer name=writer-sync-context prompt=>>WRITER_SYNC_PROMPT}}
Update .nexus/context/ documents based on the following changes.
...
<<WRITER_SYNC_PROMPT
```

- `prompt=>>IDENT` 형식으로 heredoc 시작을 선언.
- `<<IDENT` 줄에서 heredoc 종료. IDENT 종료 줄은 단독 줄이어야 한다.
- **중첩 금지**: heredoc 내부의 `{{...}}`는 매크로가 아닌 literal text로 처리.
- 한 `body.md` 내 여러 heredoc 사용 시 IDENT를 각기 다르게 지정 (`PROMPT_A`, `WRITER_PROMPT`, `RESEARCH_PROMPT` 등).

### 4 Primitive 매크로 예시

**1. `skill_activation`** (params: `skill` required, `mode` optional)

```
{{skill_activation skill=nx-plan mode=auto}}
{{skill_activation skill=nx-sync}}
```

**2. `subagent_spawn`** (params: `target_role` required, `prompt` required, `name` optional, `resume_tier_hint` optional)

```
{{subagent_spawn target_role=researcher prompt="<research question>"}}
{{subagent_spawn target_role=writer name=writer-sync-context prompt=>>WRITER_SYNC_PROMPT}}
Update .nexus/context/ documents based on the following changes.
...
<<WRITER_SYNC_PROMPT
```

**3. `task_register`** (params: `label` required, `state` ∈ {pending, in_progress, completed})

```
{{task_register label="<task label>" state=pending}}
{{task_register label="build" state=completed}}
```

**4. `user_question`** (params: `question` required, `options` required)

```
{{user_question question="Set up rules?" options=[{label: Yes, description: "proceed"}, {label: Skip, description: "later"}]}}
```

## Consumer Parser Implementation Template

각 consumer가 자기 빌드 환경(TypeScript/Node, Bun 등)에서 아래 pseudo-code를 자기 언어·tool에 맞춰 구현한다. Harness-agnostic 표현이며 실 구현 세부는 consumer 자유.

```
function expandMacros(bodyText, invocationMap, invocationsEnum):
  # 1. Heredoc boundary 추출 및 opaque 마스킹
  heredocs = extract_heredocs(bodyText)   # [{ident, content, range}]
  masked = mask_heredoc_interiors(bodyText, heredocs)

  # 2. 매크로 토큰 스캔
  macros = find_pattern(masked, /\{\{(\w+)([^}]*)\}\}/)

  # 3. 각 매크로 expand
  for m in macros:
    primitive_id, raw_params = m
    if primitive_id not in invocationsEnum:
      error "unknown primitive: " + primitive_id
    params = parse_params(raw_params)   # {key: value, ...}
    # heredoc param 해석
    for k in params:
      if params[k] starts with ">>":
        ident = params[k][2:]
        params[k] = heredocs[ident].content
    concrete = invocationMap[primitive_id].expand(params)
    replace m with concrete

  return expanded_body
```

**구현 유의사항**:

- `primitive_id`가 `invocationsEnum`에 없을 경우 silent pass 금지 — 명시적 에러 throw 필수.
- `extract_heredocs`는 `>>IDENT` 선언부터 같은 IDENT의 `<<IDENT` 종료까지 범위를 추출. 마스킹 단계에서 heredoc 내부를 opaque token으로 치환한 뒤 매크로 스캔을 수행해야 heredoc 내부의 `{{...}}`가 매크로로 오인되지 않는다.
- `invocationMap[primitive_id].expand(params)` 구현은 아래 §invocation-map.yml 템플릿 참조.

## invocation-map.yml 템플릿

Consumer repo 루트에 `invocation-map.yml` 신설. `capability-map.yml`과 평행 위상.

**claude-nexus 예시** (Claude Code 하네스):

```yaml
# invocation-map.yml — claude-nexus local resolver for nexus-core/vocabulary/invocations.yml
invocation_map:
  skill_activation:
    template: 'Skill({ skill: "claude-nexus:{{skill}}"{{, args: "{{mode}}" if mode}} })'
    # params: skill (required), mode (optional)
  subagent_spawn:
    template: 'Agent({ subagent_type: "claude-nexus:{{target_role}}"{{, name: "{{name}}" if name}}, prompt: `{{prompt}}` })'
  task_register:
    template: '{{state == "pending": TaskCreate({ subject: "{{label}}" })
               {{state == "in_progress" or "completed": TaskUpdate({ taskId: ..., status: "{{state}}" })}}'
    # state transition logic handled by expander
  user_question:
    template: 'AskUserQuestion({ question: "{{question}}", options: {{options}} })'
```

**opencode-nexus 예시** (OpenCode 하네스 — AskUserQuestion/TaskCreate native tool 부재):

```yaml
# invocation-map.yml — opencode-nexus local resolver for nexus-core/vocabulary/invocations.yml
invocation_map:
  skill_activation:
    # opencode: no live Skill() — re-emit tag for self-dispatch
    template: 're-emit tag `[{{skill}}{{: {{mode}} if mode}}]`'
  subagent_spawn:
    # opencode: hooks-based implicit routing — prompt contains target_role hint
    template: 'spawn via task tool with agent hint: {{target_role}}, prompt: `{{prompt}}`'
  task_register:
    # opencode: no TUI task tracker — no-op
    template: '(omitted — no TUI task tracker in opencode)'
  user_question:
    # opencode: no native AskUserQuestion — prose fallback
    template: 'Ask: "{{question}}"\nOptions:\n{{options | enumerate_prose}}\n(parse free-form reply)'
```

위 template 값은 예시. 실제 consumer 구현은 자기 하네스의 tool 이름·문법에 맞춰 작성.

## Rebuild Procedure

Consumer repo에서 v0.8.0을 반영하는 절차:

1. `package.json`에서 `@moreih29/nexus-core` 버전을 `^0.8.0`으로 업데이트.
2. `npm install` 또는 `bun install`로 새 버전 설치.
3. Consumer repo에 `invocation-map.yml` 신설 (위 §invocation-map.yml 템플릿 참조하여 자기 하네스 매핑 작성).
4. `harness-content/slash_command_display.md` (또는 consumer의 동등 경로) 신설:
   - claude-nexus: `/claude-nexus:nx-init` 슬래시 표기 설명
   - opencode-nexus: opencode의 해당 manual trigger 설명
5. `generate-from-nexus-core.lib.mjs`에 매크로 expander 통합 (위 §Consumer Parser Implementation Template pseudo-code 참조).
6. `bun run generate` (또는 동등 명령) 재실행 → `dist/`, `src/skills/generated/`, `skills/*/SKILL.md` 재생성.
7. `bun run validate:conformance` pass 확인 (exit code 0).
8. e2e 테스트 실행 — 매크로 expand 결과가 현 하네스에서 정상 호출되는지 확인.

**node_modules 캐시 문제 발생 시**:

```bash
rm -rf node_modules package-lock.json bun.lock
bun install
bun run generate
```

## Pitfalls

### (a) Cross-harness namespace pollution 제거

v0.7.x까지 opencode-nexus의 `src/skills/generated/*.ts`에 `claude-nexus:nx-plan`, `claude-nexus:writer` 같은 **타 하네스 namespace**가 유입되는 문제가 있었다. v0.8.0 재빌드 후 해당 산출물에 `claude-nexus:` 문자열이 0건이어야 한다. 재빌드 후 아래 grep으로 검증:

```bash
grep -rn "claude-nexus:" src/skills/generated/ || echo "✓ clean"
```

### (b) Heredoc IDENT 충돌

한 `body.md` 내에 여러 heredoc이 필요할 때 IDENT를 각기 다르게 지정해야 한다. 현재는 nx-sync에만 1개(`WRITER_SYNC_PROMPT`)가 있으나, 향후 body.md 확장 시 `PROMPT_A`, `WRITER_PROMPT`, `RESEARCH_PROMPT` 등 구별성 있는 이름을 선택. 파서는 IDENT 시작점부터 가장 가까운 동일 IDENT 종료를 찾으므로 중복 IDENT는 잘못된 boundary 파싱으로 이어진다.

### (c) `slash_command_display` harness-content 파일 누락 시

`skills/nx-init/meta.yml`의 `harness_docs_refs: [instruction_file, slash_command_display]`가 consumer repo에서 append 대상 파일을 찾지 못하면 consumer build가 실패한다. `harness-content/slash_command_display.md` 신설은 선택이 아닌 필수 단계.

### (d) 매크로 primitive_id 오타

`{{skill_activatoin ...}}`처럼 오타된 primitive_id는 consumer expander가 unknown identifier 에러로 fail해야 정상이다. Silent pass 시 런타임 버그로 이어진다 — §Consumer Parser Implementation Template pseudo-code는 명시적 에러 throw를 요구한다.

### (e) LLM prompt 내부의 hallucinated 매크로

LLM이 skill 실행 중 body.md를 읽고 `{{skill_activation skill=nx-plan}}` 패턴을 자기 output에 복제할 가능성이 있다. 이는 consumer expander 범위 밖의 LLM 행동 문제다. 별도 정책이 필요하다면 v0.9+에서 논의.
