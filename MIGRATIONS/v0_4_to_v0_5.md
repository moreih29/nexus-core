# Migration Guide: v0.4.x â†’ v0.5.0

**Release date**: 2026-04-13
**Type**: Multiple breaking changes (state schema refactor + MCP param rename + per-cycle schema_version required)
**Affected consumers**: claude-nexus, opencode-nexus, nexus-code

---

## Summary

v0.5.0 ships four breaking changes plus additive improvements:

1. `runtime.json` schema removes `plugin_version` and adds `harness_id` + `harness_version` as required fields.
2. `agent-tracker.json` schema removes `agent_type` and adds `harness_id` + `agent_name` as required fields.
3. `plan_decide` tool input parameter `summary` is renamed to `decision`.
4. `history.json` cycles gain a required `schema_version` field (per-cycle migration anchor).

All four require consumer action. This guide walks through each in order.

Additive changes (no consumer action required to pass conformance, but recommended for adoption):

- `nexus-validate-conformance` bin entry replaces the path-based `validate:conformance` script invocation.
- New `conformance/examples/plan.extension.schema.example.json` non-normative example for harness-local state extension.

## Breaking Change 1: `runtime.json` writer â€” `harness_id` + `harness_version`

### What changed

`runtime.schema.json` required array removes `plugin_version` and adds `harness_id` and `harness_version`. Consumers that write `runtime.json` at session start must update the fields they record.

### Before (v0.4.x)

```json
{
  "teams_enabled": false,
  "session_started_at": "2026-04-13T00:00:00Z",
  "plugin_version": "0.4.0"
}
```

### After (v0.5.0)

```json
{
  "teams_enabled": false,
  "session_started_at": "2026-04-13T00:00:00Z",
  "harness_id": "your-harness-id",
  "harness_version": "0.5.0"
}
```

### Consumer action required

- [ ] Session hook (runtime.json writer) update: remove the `plugin_version` field.
- [ ] Add `harness_id`: the consumer's own harness identifier string. Must match pattern `^[a-z][a-z0-9-]*$` (e.g. `"claude-nexus"`, `"opencode-nexus"`).
- [ ] Add `harness_version`: the harness plugin's version string. The value previously used for `plugin_version` can be carried over unchanged.
- [ ] Optionally add top-level `"schema_version": "0.5"` â€” this field is optional in v0.5.0 and candidate for required in v1.0.0.

### Verification

```
bunx nexus-validate-conformance
```

The `lifecycle/session-start.json` fixture asserts `harness_id` and `harness_version` are present on the written `runtime.json`. The validator exits with code 0 when both fields pass.

## Breaking Change 2: `agent-tracker.json` writer â€” `harness_id` + `agent_name`

### What changed

`agent-tracker.schema.json` items required array removes `agent_type` and adds `harness_id` and `agent_name`. The combined `"harness_id:agent_name"` string that `agent_type` previously encoded is decomposed into two discrete fields.

### Before (v0.4.x)

```json
[
  {
    "agent_type": "your-harness-id:architect",
    "agent_id": "ac01819f1cd295cb8",
    "started_at": "2026-04-13T00:00:00Z",
    "resume_count": 0,
    "status": "active"
  }
]
```

### After (v0.5.0)

```json
[
  {
    "harness_id": "your-harness-id",
    "agent_name": "architect",
    "agent_id": "ac01819f1cd295cb8",
    "started_at": "2026-04-13T00:00:00Z",
    "resume_count": 0,
    "status": "active"
  }
]
```

### Consumer action required

- [ ] Agent-tracker writer update: remove the `agent_type` concatenation logic (`"${harness_id}:${agent_name}"`).
- [ ] Record `harness_id` and `agent_name` as separate fields on every agent entry.
- [ ] `harness_id` must match `^[a-z][a-z0-9-]*$`. `agent_name` must match `^[a-z][a-z0-9-]*$` (the neutral agent id as defined in `agents/{id}/meta.yml`).
- [ ] Any read-side code that parses `agent_type` by splitting on `:` must be updated to access `harness_id` and `agent_name` directly.
- [ ] Because `agent-tracker.json` is session-scoped and deleted on session end, no migration of existing files is required. The new schema applies from the next session start.

### Verification

The `lifecycle/agent-spawn.json`, `lifecycle/agent-complete.json`, and `lifecycle/agent-resume.json` fixture assertions validate that `harness_id` and `agent_name` are present and correctly structured. Run the validator to confirm.

## Breaking Change 3: `plan_decide` parameter rename â€” `summary` â†’ `decision`

### What changed

The `plan_decide` tool's input parameter is renamed from `summary` to `decision`. The state field on `plan.json` issues has always been named `decision`; this change aligns the input parameter name with the persisted field name.

### Before (v0.4.x)

```
plan_decide({ issue_id: 1, summary: "Use declarative JSON." })
```

### After (v0.5.0)

```
plan_decide({ issue_id: 1, decision: "Use declarative JSON." })
```

### Consumer action required

- [ ] MCP tool wrapper for `plan_decide`: rename the input parameter from `summary` to `decision`.
- [ ] Any handler that processes the `[d]` tag and forwards a `summary` key to `plan_decide` must be updated to pass `decision` instead.
- [ ] Prompt templates or documentation that reference the `summary` parameter by name for `plan_decide` must be updated to `decision`.
- [ ] Optional parameters `how_agents`, `how_summary`, and `how_agent_ids` are unchanged.

### Verification

All three fixtures in `conformance/tools/plan-decide.json` use `"decision"` in `action.params` and in `uncovered_params`. The conformance runner passes only when the tool implementation accepts `decision` and ignores `summary`.

## Breaking Change 4: `history.json` per-cycle `schema_version` required

### What changed

`history.schema.json` adds `schema_version` to the `cycles[]` items required array. Every cycle written to `history.json` from v0.5.0 onward must include this field. The file-level `schema_version` field remains optional in v0.5.0.

Value format: `"\d+\.\d+"` â€” for cycles written under v0.5.0, use `"0.5"`.

### Before (v0.4.x) â€” `schema_version` absent, no validation failure

```json
{
  "cycles": [
    {
      "completed_at": "2026-04-13T00:00:00Z",
      "branch": "main",
      "plan": { "id": 1, "topic": "...", "issues": [], "created_at": "2026-04-12T00:00:00Z" },
      "tasks": []
    }
  ]
}
```

### After (v0.5.0) â€” `schema_version` required on each cycle

```json
{
  "schema_version": "0.5",
  "cycles": [
    {
      "schema_version": "0.5",
      "completed_at": "2026-04-13T00:00:00Z",
      "branch": "main",
      "plan": { "id": 1, "topic": "...", "issues": [], "created_at": "2026-04-12T00:00:00Z" },
      "tasks": []
    }
  ]
}
```

### Consumer action required

- [ ] `task_close` implementation: when archiving a cycle to `history.json`, include `"schema_version": "0.5"` on the new cycle object.
- [ ] Optionally record `"schema_version": "0.5"` at the file level as well (recommended for future readers; optional in this release).
- [ ] Existing cycles in `history.json` written before v0.5.0 will not have `schema_version`. The required constraint is enforced only on newly written cycles. Existing cycles are not retroactively invalidated by the schema, but consumers that run full-file validation may need to relax validation for the legacy cycles or backfill them (see Troubleshooting).

### Verification

The `task-close.json` fixture postcondition asserts that the archived cycle includes `"schema_version": "0.5"`. The `history.schema.json` validation fails for any new cycle missing this field.

## Additive: `nexus-validate-conformance` bin entry

### What changed

`package.json` adds a `bin` entry `{"nexus-validate-conformance": "./scripts/conformance-coverage.ts"}` and includes `scripts/` in the `files` array so the script is distributed in the npm tarball.

### Before (v0.4.x)

```
# The scripts/ directory was not included in the tarball.
bun run validate:conformance   # worked only inside the nexus-core repo itself
```

### After (v0.5.0)

```
bunx nexus-validate-conformance
# or
npx nexus-validate-conformance
```

### Consumer action required

- [ ] Update CI pipeline: replace any `bun run validate:conformance` or direct path invocation (e.g. `node_modules/@moreih29/nexus-core/scripts/...`) with `bunx nexus-validate-conformance`.
- [ ] No code changes required â€” this is an invocation-path change only.

## Upgrade Protocol

1. Update the dependency pin:
   ```
   bun add --dev @moreih29/nexus-core@^0.5.0
   ```
2. Apply Breaking Change 1: update the session hook that writes `runtime.json`.
3. Apply Breaking Change 2: update the agent-tracker writer and any read-side code that parsed `agent_type`.
4. Apply Breaking Change 3: rename the `summary` parameter to `decision` in the `plan_decide` tool wrapper and any tag handler that calls it.
5. Apply Breaking Change 4: update the `task_close` implementation to record `schema_version` on each new history cycle.
6. Run `bunx nexus-validate-conformance` and confirm all fixtures pass (exit code 0).
7. Update your CI pipeline's conformance gate step to use the bin-based invocation.
8. If you maintain custom fixtures, verify they cover any newly required fields.

## Troubleshooting

### `bunx nexus-validate-conformance` exits with ENOENT

Re-run `bun install` (or `bun i`) to regenerate bin symlinks in `node_modules/.bin/`. This is the most common cause after a fresh install on a new machine.

### `agent-tracker.json` entries from a previous session still have `agent_type`

`agent-tracker.json` is deleted at session end. No migration of existing files is required. The new schema is applied automatically from the next session start.

### Existing `history.json` cycles have no `schema_version`

The required constraint is write-time only. Cycles written before v0.5.0 are not retroactively invalid. If your tooling runs full-file schema validation on read (e.g. in a CI read-back check), either:
- Exempt legacy cycles from the `schema_version` required check by filtering on `completed_at` before a known cutoff date, or
- Backfill legacy cycles: add `"schema_version": "pre-0.5"` (or an estimated version) to each existing cycle before the validation step. This is a one-time migration script; the value is informational only.

The validator enforces `schema_version` only when a new cycle is archived, not when reading the full file.

### `harness_version` value format

`harness_version` is a free string with no pattern constraint in the schema. Semver is recommended for consistency with standard package versioning, but is not required.

### `plan_decide` calls returning errors after the rename

If the tool implementation still reads from a `summary` key, calls using `decision` will appear to succeed (parameter not found) but will write an empty decision string. Confirm the tool wrapper reads from `decision`, not `summary`, and re-run the `plan-decide.json` fixtures to verify.
